## Diagnosis

**Recall.ai webhook dashboard showed 62.5% error rate** — webhooks reaching `https://app.kolasys.ai/api/webhooks/recall` but returning 400.

Investigated the webhook route (`src/app/api/webhooks/recall/route.ts`):
- Code was already correct: `request.text()` for raw body, Svix verify before JSON.parse
- Confirmed reachable: plain POST returns 400 "Invalid signature" (not 404, not 500)
- Confirmed secret is set on Vercel: endpoint returns 500 "Webhook secret not configured" only when env var is absent
- **Root cause**: `RECALLAI_WEBHOOK_SECRET` on Vercel (`whsec_ptHu…`) does not match the signing secret currently configured in the Recall.ai/Svix dashboard. Recall.ai's webhooks are signed with a rotated or re-provisioned secret.

Evidence: a test webhook signed with our local `whsec_ptHu…` hits production and returns **200 OK**. Real Recall.ai deliveries at 19:31 UTC returned **400**. Therefore Recall.ai is signing with a different key.

## Bug Found: Neon HTTP Adapter — DateTime Comparison Unreliable

While building the bot-poller, the worker ran but logged "found 0 stuck" even though 3 clearly-stuck recordings existed.

**Root cause**: The Neon HTTP adapter has a timezone offset bug. Prisma `DateTime: { lt: new Date(...) }` comparisons return 0 rows even when records are obviously older. Example:

```ts
// Records have createdAt = 15:22, 15:24, 15:52 UTC
// cutoff = 17:52 UTC (now - 5 min)

createdAt: { lt: cutoff }   // → 0 results ← BUG
createdAt: { lt: new Date('2026-06-08T20:00Z') }  // → 3 results ← fine
createdAt: { lt: new Date('2026-06-08T16:00Z') }  // → 0 results ← still wrong
```

JavaScript comparison `rec.createdAt < cutoff` worked correctly for all three records.

**Fix applied**: Removed the `createdAt` filter from the Prisma `where` clause and applied it in JavaScript after fetching:

```ts
// BEFORE (broken)
const stuck = await db.recording.findMany({
  where: { botId: { in: doneBotIds }, status: 'PROCESSING', s3Key: null,
           createdAt: { lt: new Date(Date.now() - STUCK_THRESHOLD_MS) } },
})

// AFTER (fixed)
const candidates = await db.recording.findMany({
  where: { botId: { in: doneBotIds }, status: 'PROCESSING', s3Key: null },
  select: { id: true, botId: true, title: true, createdAt: true },
})
const now = Date.now()
const stuck = candidates.filter(rec => now - rec.createdAt.getTime() > STUCK_THRESHOLD_MS)
```

**Rule going forward**: Never use Prisma `DateTime` comparisons (`lt`, `gt`, `gte`, `lte`) in where clauses on Neon. Always fetch and filter in JavaScript.

## Bot Poller Worker — Built and Deployed

### What it does

`src/workers/bot-poller.worker.ts` — polls Recall.ai every 2 minutes:
1. Fetches all bots from `GET /api/v1/bot/?created_at_after=<24h ago>&limit=100`
2. Filters to bots whose last `status_changes` entry is `done` or `call_ended`
3. Queries DB: recordings with `botId IN doneBotIds AND status=PROCESSING AND s3Key=null`
4. Age check (>5 min) in JavaScript (not SQL — Neon bug)
5. Enqueues to `botIngestionQueue` for each stuck recording
6. Heartbeat log every 30 polls (~1 hour)

**Architecture decision**: webhooks are now an optimization, not the critical path. Every stuck MEETING_BOT recording auto-recovers within ≤2 minutes.

### Deployment to Railway

Service: `bot-poller-worker` in `glorious-serenity` project.

**Key deployment gotchas encountered:**
- `railway add --service bot-poller-worker` creates an empty service (no GitHub)
- `railway up` with an empty service triggers a full Next.js build (Nixpacks auto-detects Next.js) — WRONG for workers, build fails because `RESEND_API_KEY` and other vars not present
- `NIXPACKS_BUILD_CMD=npm install` env var did NOT override the Next.js detection
- **Fix**: used Railway GraphQL API to connect service to GitHub repo:
  ```
  mutation { serviceConnect(id: "bcf7506d...", input: { repo: "kolasystems/kolasys-ai", branch: "main" }) }
  mutation { serviceInstanceDeploy(serviceId: "bcf7506d...", environmentId: "67d64227...", latestCommit: true) }
  ```
  GitHub-connected deploy reads `railway.toml` correctly and uses the service-specific `startCommand`.

**`railway.toml` entry added:**
```toml
[[services]]
name = "bot-poller-worker"

[services.build]
builder = "NIXPACKS"

[services.deploy]
startCommand = "npx tsx src/workers/bot-poller.worker.ts"
restartPolicyType = "always"
```

**Env vars on `bot-poller-worker`:** `DATABASE_URL`, `REDIS_URL`, `RECALLAI_API_KEY`, `SENTRY_DSN`, `NEXT_PUBLIC_APP_URL`

**Confirmed running:**
```
[bot-poller] worker starting…
[bot-poller] checked 16 bots (16 done), found 0 stuck, enqueued 0
```
